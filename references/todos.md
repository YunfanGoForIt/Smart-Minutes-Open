# 待办（# 待办）

从文字稿中提取**会议中明确指派的行动项**，写成复选框：

```
- [ ] {动作：动词开头，明确做什么} <cite type="user" user-name="{责任人}"></cite>
```

## 提取规则

- **只提取真正的行动项**：必须满足"有人被指派/认领 + 具体动作 + 可辨识的目标或时间"。三者缺一不可。
- **优先捕捉承诺性表述**：留意"我回去发给你""我跟 XX 沟通一下""后面画一下""我来负责 X"这类**第一人称承诺/认领**——这些是最可靠的待办来源，逐条记录并标注责任人。
- **不要**把"建议大家……""可以考虑……""以后要……"这类**建议性/倡议性**表述转成待办——尤其是分享类、圆桌讨论类会议，发言者的个人建议不是会议行动项。
- **责任人必须逐字确认归属**：谁认领/被指派，就从文字稿原文逐字核对——是"我回去发给你"则责任人是说话人，是"XX 你来负责"则责任人是 XX。**严禁张冠李戴**（如把 A 说要做的写成 B 负责）。拿不准归属就省略 `<cite>`，不要瞎填。
- **区分"行动项"与"日程/安排"**：只有**需要某人主动执行的动作**（联系/核实/整理/开启/提交/发送/绘制/沟通）才列入待办；**被动日程**（"下周一考试""按既定顺序报告""明天开会"）不算待办，应放进要点梳理的时间安排板块。
- **捕捉执行性待办**：除显式承诺外，留意"画一下跟团队讨论""跟 XX 沟通对接""让对方再改""整理清单"等**动词+明确对象+负责人**的执行性表述——这些是待办，别漏。
- **待办时间以最终确认为准**：若截止时间在会议中被改动（如"原定今天，缓至明晚"），待办写**最后一次确认的截止时间**，不要写已被推翻的旧时间。
- **区分"决策"与"行动项"**：行动项（"周总当经纪人联系商演""workshop 内容由 X 负责"——有明确执行人+动作）应归入待办，不要放进关键决策。
- **待办严防过度生成**：只收录文字稿中**明确指派给某人**的行动项（负责人+可执行动作+明确产出/目标）。**排除**泛泛建议（"可以拿个证书""应该搞 workshop"——若未指派给具体人则不入待办）、身份介绍、课程内容说明。拿不准是不是被指派了，就不入待办。
- 责任人能从上下文明确推断则填：`<cite type="user" user-id="USER_ID_PLACEHOLDER" user-name="{人名}"></cite>`（user-id 用占位符 `USER_ID_PLACEHOLDER`，因为生成时拿不到真实飞书 user-id；人名用文字稿出现的名字）。无法确定责任人则省略 `<cite>`。
- 没有明确行动项就**整节省略**（连同其 readonly-block），不要为了凑数编造 todo。
- 每条聚焦一个动作，别把一坨事塞进一条。
