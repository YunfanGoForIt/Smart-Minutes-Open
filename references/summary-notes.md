# 要点梳理（# 总结 正文）— grid 多列结构化

紧跟画板标记，先写**一句引导文字**概括会议围绕什么展开、得出了什么，以"内容如下："结尾。例如：

> 本次会议围绕社区发展规划、组织架构、项目安排以及合作事宜展开讨论，明确了社区发展方向和近期工作重点，……内容如下：

然后用 **`<grid>` + `<column>` 多列布局**组织要点（这是飞书纪要最显著的格式特征）：

```
<grid>
<column width-ratio="0.750000">
- **{一级要点}**

  - **{二级要点}**
  
    - {三级要点：一句具体说明}
    - {三级要点}
  - **{二级要点}**
  
    - {三级要点}
</column>
<column width-ratio="0.250000">
{副列：对该要点的图示/表格的文字描述，或留空}
</column>
</grid>
```

## 要点梳理规则

- 一级要点用 `- **加粗**`，二级要点缩进两空格同样加粗，三级要点再缩进、不加粗、写具体内容。
- **`<grid>` 的用途是"局部文字+图片并排"**：仅当某一段要点需要配一张图（图说/示意/表格图）时，才把该段要点放进主列 `<column>`、图片放副列 `<column>`。**不要用 `<grid>` 包裹整个总结**——大部分要点直接用普通列表即可，原版飞书纪要也是局部使用 grid。
- **grid 数量对齐原版**：若会议有"核心架构/产品矩阵/方案示意"类内容 + "后续工作计划/时间线/任务表"类内容，应分别配 2 个 grid（主列文字要点 + 副列 `IMAGE_PLACEHOLDER` 图），与原版双 grid 结构对齐；只有一个图示内容时才用 1 个 grid。
- **正文对比表必须保留 grid**：若某要点段是对多个对象/方法/站点的**并列对比**（如"找对标竞品"对比多站点、"找早期用户"对比多渠道），**必须用 `<grid>` + `<column>` 还原对比表形态**（主列放对比项、副列放图占位），**不得退化为纯 bullet 列表**。
- 主列 `width-ratio` 通常 0.55-0.85，副列 0.15-0.45。副列图片用 `![图片：{对该图内容的一句话描述}](IMAGE_PLACEHOLDER)`——**url 一律用占位符 `IMAGE_PLACEHOLDER`**，绝不编造假域名/假链接；无图就别加 grid，直接列表。
- **覆盖面要与原版对齐**：除了核心议题，文字稿中明显存在的**生活向/闲聊/工具/杂项话题**也应作为 1-2 个次要主题简要记录（如"生活与消费讨论""技术与工具交流"），不要为聚焦核心而砍掉这些——原版会保留它们以完整反映会议。
- **讲座/教学类要点的叙事顺序**：若主讲人按"第一块/第二块/第X章/第X个PPT"分块推进，要点应**保留这个分块顺序**（可在块内做主题归纳，但块标题用"第一块：…""第二块：…"），**不要跨块重组**为后置分类（如"图像基础/恢复/分割"），以免偏离原版叙事。
- **讲座类要点不得压缩过狠**：复习/串讲/教学类会议的每个知识块，应保留原版强调的**核心概念名+一句话解释**，每个子要点都要有（如"同态滤波：光源×反射→log 分离→偏高通压低频，强光下车窗看不清例"），**不要压缩成每块 2-3 条短 bullet**。目标是覆盖原版 ~80% 的子要点，宁可多列几条也不要丢考点。
- 一个一级要点下可嵌套多个 `<grid>`，把不同子主题分别做成两列。
- 要点必须**忠实于文字稿**：原文提到的才写，提炼而非扩写；原文没有的人名/数据/结论绝不编造。
- 层级不超过三级；每条三级要点一句话，避免长段。
- **要点精炼但有边界**：每个一级区块的子项不超过 5 条，超过的细节下沉到智能章节摘要或省略，数据/案例挑最典型 1-2 个。**但精炼≠丢块**——要点块数应为原版/会议实际主题数的 50%-80%，**定调性内容**（个人经历/创业理念/背景铺垫等）至少保留 1 个子块或并入相关块首条，不得整块抹去。低于 50% 视为信息丢失。
- 要点之间逻辑分组清晰（按主题/时间/事项），不要平铺堆砌。
